# MDEB App

Self-hosted personal data manager with an outdoors/hiking + cartography + technology aesthetic.

## Stack

| Layer | Technology |
|-------|-----------|
| Frontend | Vue 3 + TypeScript + TailwindCSS + Vite + Pinia |
| Backend | FastAPI (Python 3.12) + SQLAlchemy async + Alembic |
| Database | PostgreSQL 16 |
| Deployment | Docker + docker-compose |

## Features

- **Task Management** — full CRUD for tasks with categories, projects, priority, effort, and scheduled dates
  - Statuses: `TODO`, `IN_PROGRESS`, `WAITING`, `COMPLETE`, `BACKLOG`
  - Priority: `LOW`, `MEDIUM`, `HIGH`, `URGENT`
  - Effort: `LOW`, `MEDIUM`, `HIGH`
  - Timestamped notes per task; cascade-deletes with parent task

- **Journal + Day Rating** — daily markdown journal with auto-save, 1–5 star day rating, mini date-picker with entry indicators, and "On This Day" sidebar
  - EasyMDE editor with read/write toggle
  - Upsert-by-date pattern (one entry per date)

- **Time Tracking** — ledger-style day-by-day log with month navigation, category/subcategory management, and monthly summary sidebar
  - 15-minute granularity; retroactive entry only (no live timer)
  - Overnight entries supported (`end_time` can span to the next calendar day)
  - Non-blocking overlap detection
  - 6 default categories seeded: Work, Health, Social, Leisure, Life Admin, Transit

- **Notes** — markdown note-taking with hierarchical folder organization
  - Nested folders with re-parenting on delete
  - Unfiled notes support
  - Full EasyMDE markdown editor with autosave

- **Finance** — personal finance ledger with accounts, transactions, merchants, and monthly summaries
  - Account types: debit, credit, investment, other
  - Transaction types: expense (credit), income (debit), internal transfer
  - Monthly account value snapshots for reconciliation
  - Profit calculation across profit-eligible accounts
  - Merchant management with archive support

- **Timeline** — calendar overview aggregating signals from all modules
  - Per-day indicators: journal entry, transactions, completed tasks, time entries
  - Click any day to open a full day profile panel

## Prerequisites

- [Docker](https://docs.docker.com/get-docker/) and [Docker Compose](https://docs.docker.com/compose/install/)

## Development

### 1. Configure environment

```bash
cp .env.example .env
```

Edit `.env` and set your credentials:

```env
POSTGRES_USER=mdeb
POSTGRES_PASSWORD=your-secure-password
POSTGRES_DB=mdeb
DATABASE_URL=postgresql+asyncpg://mdeb:your-secure-password@db:5432/mdeb
SECRET_KEY=a-long-random-secret-string
ALLOWED_ORIGINS=http://localhost:5173
VITE_API_BASE_URL=http://localhost:8000
```

### 2. Start services

```bash
docker compose up --build
```

| Service | URL |
|---------|-----|
| Frontend (Vite dev server) | http://localhost:5173 |
| Backend API | http://localhost:8000 |
| API health check | http://localhost:8000/api/health |
| Interactive API docs | http://localhost:8000/docs |

Both the frontend and backend mount local source directories as volumes, so changes are reflected without rebuilding.

### 3. Run database migrations

On first run (or after model changes), apply Alembic migrations inside the backend container:

```bash
docker compose exec backend alembic upgrade head
```

To generate a new migration after changing SQLAlchemy models:

```bash
docker compose exec backend alembic revision --autogenerate -m "describe your change"
docker compose exec backend alembic upgrade head
```

> **Note:** Adding a value to an existing PostgreSQL enum cannot be autogenerated. Write the migration manually using `op.execute("ALTER TYPE <name> ADD VALUE IF NOT EXISTS '<value>'")`.

## Production

For production, you'll want to:

1. Set strong, unique values for `POSTGRES_PASSWORD` and `SECRET_KEY` in `.env`.
2. Update `ALLOWED_ORIGINS` to your production domain(s).
3. Update `VITE_API_BASE_URL` to your production API URL.
4. Build and run with compose (no hot-reload):

```bash
docker compose up --build -d
```

For a production-grade setup, consider placing a reverse proxy (e.g. Nginx or Caddy) in front of both services to handle TLS and serve the built frontend as static files instead of the Vite dev server.
